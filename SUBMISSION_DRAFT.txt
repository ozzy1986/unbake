UNBAKE: API ДЛЯ ТЕКСТА ПЕСНИ С ТАЙМКОДАМИ

1) Коротко о задаче

Нужно сделать API, которое принимает ссылку на вокальную дорожку и возвращает текст песни, синхронизированный по времени.

Вход:
- S3 presigned URL на vocal stem;
- формат файла: m4a, 256 kbps;
- вокал получен через Demucs v4;
- внутри вокала могут быть lead и backing vocals, но разделять их сейчас не нужно.

Выход:
- обычный текст песни;
- текст с таймкодами строк, похожий на LRCLIB/LRC;
- желательно таймкоды по словам, потому что это лучше для karaoke-style UX.

Главные критерии:
1. Accuracy: минимум ошибок и никаких случайных галлюцинаций.
2. Cost: меньше 0.05$ за 3 минуты трека.
3. Latency: пользователь iOS-приложения не должен ждать слишком долго.


2) Что мы сделали

Мы собрали рабочий прототип пайплайна и реально прогнали его на вокальных stem-файлах.

Пайплайн:
- берём m4a vocal stem;
- прогоняем через ASR-модель;
- получаем текст, строки с таймкодами и слова с start/end;
- сравниваем результат с найденным reference lyrics;
- считаем автоматические метрики CER и WER;
- отдельно проверяем второй проход alignment для уточнения таймкодов.

Код лежит в репозитории:
https://github.com/ozzy1986/unbake

Основные файлы:
- tools/transcribe.py - распознаёт аудио и сохраняет JSON с plainLyrics, syncedLyrics, segments, words, runtime;
- tools/evaluate_text.py - считает CER и WER;
- tools/align_whisperx.py - делает второй проход alignment через WhisperX;
- tools/run_multilingual_baseline.py - запускает baseline по нескольким языкам;
- data/ - входные аудио и reference lyrics;
- outputs/ - реальные результаты прогонов.


3) Схема решения

[iOS app]
    |
    | POST /v1/transcriptions
    | audioUrl = S3 presigned URL
    v
[API]
    |
    | создаёт async job
    v
[Worker]
    |
    +--> скачивает m4a
    +--> проверяет размер, тип, длительность
    +--> конвертирует во внутренний формат mono 16k wav
    +--> ASR: распознаёт текст
    +--> alignment: уточняет таймкоды слов
    +--> форматирует результат как LRCLIB-like JSON
    v
[Result storage]
    |
    | GET /v1/transcriptions/{jobId}
    v
[iOS app получает lyrics + timestamps]


4) Формат API

API лучше делать асинхронным, потому что обработка трека может занимать десятки секунд.

POST /v1/transcriptions

Пример запроса:

{
  "audioUrl": "https://s3-presigned-url...",
  "languageHint": "ru"
}

Пример ответа:

{
  "jobId": "uuid",
  "status": "queued"
}

GET /v1/transcriptions/{jobId}

Пример успешного ответа:

{
  "id": "uuid",
  "status": "succeeded",
  "duration": 218.03,
  "plainLyrics": "...",
  "syncedLyrics": "[00:14.61] ...",
  "words": [
    {
      "word": "example",
      "start": 14.61,
      "end": 15.02,
      "confidence": 0.87
    }
  ],
  "meta": {
    "language": "en",
    "asrModel": "faster-whisper-small",
    "alignment": "whisperx"
  }
}

Формат близок к LRCLIB, но дополнен words, потому что для нашего продукта важны таймкоды по словам, а не только по строкам.


5) Формат аудио

Снаружи принимаем m4a 256 kbps, как указано в задании.

Внутри лучше сразу приводить аудио к:
- wav;
- mono;
- 16 kHz;
- PCM.

Причина простая: ASR и alignment-модели стабильнее работают с таким форматом. m4a хорош для передачи и хранения, но не как рабочий формат внутри ML-пайплайна.


6) Что показали реальные прогоны

Тестировали на слабом локальном железе:
- CPU: Intel i5-7300HQ;
- RAM: 8 GB;
- GPU: GTX 1050 4 GB, но основной baseline был CPU.

Модель: faster-whisper small.

Результаты:

| Язык/трек | CER | WER | Время обработки |
| --- | ---: | ---: | ---: |
| EN rockstar | 0.250 | 0.345 | 77s на 218s аудио |
| ES BELLAKEO | 0.612 | 0.783 | 381s на 197s аудио |
| FR Place de la Republique | 0.228 | 0.391 | 62s на 261s аудио |
| RU Полковнику никто не пишет | 0.245 | 0.604 | 198s на 243s аудио |

Вывод:
- пайплайн работает end-to-end;
- текст, syncedLyrics и word timestamps получить можно;
- но weak CPU + small model не дают production-quality результата.

Основная проблема не в API, а в качестве ASR на грязном вокале после Demucs.


7) Что дал второй проход alignment

Мы проверили WhisperX alignment на EN sample.

Результат:
- текстовые метрики не изменились;
- таймкоды слов изменились и стали более согласованными с аудио.

Это ожидаемо:
- ASR отвечает за "какие слова распознаны";
- alignment отвечает за "в какое время эти слова звучат".

Если ASR уже неправильно распознал слово, aligner обычно не исправляет текст. Он только точнее привязывает уже имеющийся текст ко времени.


8) Как оценивали качество

Мы написали Python-оценку:
- нормализуем текст;
- считаем CER - ошибка по символам;
- считаем WER - ошибка по словам;
- сравниваем output модели с reference lyrics.

Этого достаточно для первого честного сравнения моделей.

Что стоит добавить дальше:
- hallucination rate: сколько лишнего текста модель придумала;
- missing words rate: сколько слов пропущено;
- timestamp error: средняя ошибка таймкода, если есть word-level reference;
- real-time factor: время обработки / длительность аудио.


9) Alternatives considered

A) Managed ASR API: OpenAI, Deepgram, AssemblyAI

Плюсы:
- быстрее всего запустить;
- не нужно обслуживать GPU;
- можно использовать как benchmark качества.

Минусы:
- дорого на масштабе;
- меньше контроля над ошибками;
- не всегда понятно, как модель ведёт себя на музыкальном вокале, а не обычной речи.

Примерная стоимость для 3-минутного трека:
- около 0.009-0.023$ за запрос.

Для 100 запросов в день это примерно 27-69$ в месяц.
Для 10000 запросов в день это уже тысячи долларов в месяц.


B) Self-hosted ASR + alignment

Плюсы:
- лучше контролируем стоимость;
- можно оптимизировать под vocal stems;
- дешевле на масштабе;
- больше контроля над quality gates и post-processing.

Минусы:
- нужен GPU;
- нужно самим следить за latency, очередью и стабильностью;
- нужен benchmark на более сильных моделях.

Целевой вариант для production: self-hosted GPU worker.


C) Lyrics-first через ShazamKit/LRCLIB

Плюсы:
- очень дёшево и точно для известных оригинальных треков;
- на iOS ShazamKit можно использовать бесплатно;
- если трек точно найден, можно взять lyrics и сделать alignment.

Минусы:
- не подходит как единственный способ;
- пользователи могут загружать каверы, перепевки и версии с другим текстом;
- если слепо взять текст из базы, можно вернуть красивый, но неправильный результат.


10) Выбранный подход

Лучший подход - hybrid.

1. ASR-first для кастомных вокалов и каверов.
   Это основной путь, потому что текст может отличаться от оригинала.

2. ShazamKit/LRCLIB как помощник для известных оригинальных треков.
   Если iOS уверенно определил трек, можно использовать найденный lyrics как подсказку или reference для alignment.

3. Alignment после ASR или reference lyrics.
   Нужен для точных таймкодов по словам.

Главный принцип: lyrics database не должна быть единственным source of truth.


11) Стоимость

Цель задания: меньше 0.05$ за 3 минуты трека.

Managed API:
- подходит для маленького старта;
- при 100 запросах в день может уложиться примерно в 27-69$ в месяц;
- при 10000 запросах в день становится слишком дорогим.

Self-hosted GPU:
- требует аренды или своего GPU;
- зато лучше масштабируется по стоимости;
- реалистичная цель при нормальном GPU и оптимизации: примерно 0.001-0.01$ за 3-минутный трек.

Из-за ограничений проекта мы не запускали платный GPU benchmark. Поэтому разделяем:
- реальные локальные измерения;
- теоретическую production-оценку по публичным ценам и ожидаемой скорости GPU-инференса.


12) На каком железе крутить

API/backend:
- обычный VPS.

ASR/alignment:
- отдельный GPU worker.

Для старта на 100 запросов в день:
- можно использовать on-demand/batch GPU worker, чтобы не платить за простой;
- либо временно managed ASR API, если важнее быстро проверить продукт.

Для роста:
- self-hosted GPU выгоднее.

CPU-only не подходит для production качества и latency. Он подходит только для разработки и дешёвых smoke tests.


13) Что может пойти не так

1. Demucs даёт не чистый вокал.
   В stem остаются артефакты, инструменты, бэки, наложения голосов.
   Это ухудшает ASR.

2. Пение и рэп сложнее обычной речи.
   Быстрый флоу, сленг, адлибы и эффекты сильно повышают ошибку.

3. Галлюцинации.
   Модель может придумать слова.
   Нужны confidence thresholds, повторные проверки, language consistency и фильтры подозрительных повторов.

4. Каверы и изменённый текст.
   Нельзя слепо брать lyrics из базы.
   ASR должен оставаться основным источником для кастомного вокала.

5. Presigned URL может истечь.
   Нужны таймауты, понятные ошибки и retry policy.


14) Итог

Мы не просто описали архитектуру, а собрали и запустили рабочий прототип.

Что доказано:
- vocal m4a можно обработать end-to-end;
- можно получить plainLyrics, syncedLyrics и word timestamps;
- есть автоматическая оценка качества;
- есть реальные outputs и метрики;
- понятно, где главный bottleneck.

Главный вывод:
- текущий CPU baseline на small model не годится для production accuracy;
- для качественного результата нужна более сильная ASR-модель на GPU;
- alignment нужен, но он решает задачу тайминга, а не исправления текста.

Финальная рекомендуемая архитектура:
- async API;
- ASR-first pipeline;
- optional ShazamKit/LRCLIB для известных оригиналов;
- forced alignment для точных word timestamps;
- self-hosted GPU worker как целевой production-вариант по стоимости.

Это честный результат с учётом ограничений:
1. денег на платные сервисы/GPU не было;
2. локальное железо слабое.

Мы не стали делать вид, что baseline уже решает задачу идеально. Вместо этого мы построили проверяемый прототип, получили реальные цифры и показали, какой следующий шаг нужен для production-quality результата.
