UNBAKE TEST TASK - API FOR TIME-SYNCED LYRICS

1) What we had to build

We needed an API that accepts an S3 presigned URL to a vocal stem and returns song lyrics synchronized by time.
Input audio is a vocal stem after Demucs v4 (m4a, 256 kbps).
Priority: Accuracy first, then Cost, then Latency.


2) What we actually implemented and tested

We built and ran a real prototype pipeline:

- Input: vocal stem (m4a)
- ASR pass: speech-to-text with multilingual model
- Sync output: line-level + word-level timestamps
- Evaluation: automatic metrics against reference lyrics

Implemented scripts:

- tools/transcribe.py
  Converts audio to transcription output JSON:
  - plainLyrics
  - syncedLyrics
  - segments
  - words
  - runtime

- tools/evaluate_text.py
  Calculates CER and WER from model output vs reference lyrics.

- tools/align_whisperx.py
  Second pass alignment (WhisperX) for better timestamp precision.

- tools/run_multilingual_baseline.py
  Batch baseline run on available dataset subset.

4) Data and language coverage we used

We used the dataset provided in this task context (local project data).
Important: available set in our test run was not full 8-language target from the brief.
So our measured results are honest for the files we had and processed, not a fake "all languages passed" claim.


5) Main measured results (real runs)

On local weak hardware baseline (CPU-first), pipeline works end-to-end, but text quality is not production-ready.

Example measured quality:

- EN sample (small model): CER ~0.250, WER ~0.345
- RU sample (small model): high word-level error
- ES sample (small model): very high error on noisy stem

Conclusion from real runs:

- We can produce text + timestamps reliably as a process.
- We cannot claim production-quality lyrics text with current weak baseline model/hardware.


6) What second pass alignment changed

We tested a second pass (WhisperX alignment).
Result:

- It improves/changes timestamp placement.
- It does NOT fix wrong recognized words from ASR.

This is expected: ASR decides "what text", aligner decides "where in time".


7) Input format decision

Task input is m4a 256 kbps and we keep it as external input format.
Internally, best practice is to convert to mono 16k PCM wav before ASR/alignment.
Reason: stable decoding and better model compatibility.


8) API design (simple, practical)

POST /v1/transcriptions
- input: audioUrl (presigned), optional languageHint
- output: jobId, status=queued

GET /v1/transcriptions/{jobId}
- output fields:
  - plainLyrics
  - syncedLyrics (LRC-like)
  - words: [{word, start, end, confidence}]
  - model and runtime metadata

This format is close to LRCLIB but extended with word-level timing.


9) Alternatives considered

A) Managed ASR APIs
Pros: easy and fast.
Cons: expensive at scale, lower control.

B) Self-hosted ASR + aligner
Pros: best long-term cost control and customization.
Cons: needs GPU operations and careful tuning.

C) Lyrics-first from track ID only
Pros: cheap for known originals.
Cons: fails for covers/custom rewritten vocals.

Chosen direction: hybrid
- ASR-first for custom/covers
- optional track-identity/lyrics lookup as helper, not source of truth.


10) Cost estimate (theoretical, with no paid runs)

Managed APIs are usually acceptable at ~100 requests/day but expensive near 10k/day.
Self-hosted GPU is usually better long-term for cost per 3-minute track.
The target constraint (< $0.05 per 3 min) is realistic with efficient self-hosted setup.

Since we had no budget, we provide theory + local measured baseline, and clearly separate estimates from measured production-scale numbers.


11) Risks and mitigations

Risk: Demucs stem artifacts and overlapping vocals
Mitigation: stronger ASR model + confidence filtering + robust preprocessing

Risk: Hallucinated text
Mitigation: quality gates, confidence thresholds, language consistency checks

Risk: Covers with changed lyrics
Mitigation: do not trust external lyrics blindly; ASR remains source of truth

Risk: presigned URL expiry/failure
Mitigation: strict download timeout/retry and explicit API errors


12) Final honest conclusion

What is proven:

- Clear understanding of why current baseline fails on quality

What is still needed for production claim:

This is an honest, test-backed outcome under constraints:
